Отчет о выполнении задачи по умножению матриц

ВЫЧИСЛИТЕЛЬНОЕ ОКРУЖЕНИЕ
---------------------------
Процессор:            Intel Core i7-1260P
Кэш L1d (P-core):     48 KiB (8-way associative)
Кэш L2 (P-core):      1.25 MiB
ОС:                   Linux Mint
Компилятор:           g++ (GCC) 13.4.0
Флаги компиляции:     -O3 -ffast-math -march=native -funroll-loops
Векторное расширение: AVX2 / FMA3 (AVX-512 отсутствует на данной модели CPU)
GPU:                  Intel Iris Xe Graphics (96 EUs) через OpenCL

ТЕОРЕТИЧЕСКАЯ ОЦЕНКА ЭФФЕКТИВНОСТИ
-------------------------------------
Для архитектуры Intel Golden Cove (мой 1260P-core) расчет пиковой производительности:
- Ширина вектора AVX2 (fp32): 8 элементов.
- Количество портов исполнения FMA: 2 порта.
- Операций за одну инструкцию  FMA: 2 (умножение + сложение).

Пиковая производительность FLOPS/cycle:
8 (float) * 2 (порты) * 2 (ops) = 32 FLOPS/cycle.


ПОДРОБНЫЙ АНАЛИЗ ЭВОЛЮЦИИ ОПТИМИЗАЦИИ МАТРИЧНОГО УМНОЖЕНИЯ
----------------------------------------------------------------------
Процесс оптимизации был направлен на преодоление Memory Wall
и максимальную утилизацию вычислительных ресурсов процессора.

ЭТАП 1: НАИВНАЯ РЕАЛИЗАЦИЯ (IJK) И ПРОБЛЕМА КЭША
------------------------------------------------
Классическое тройное вложенное циклическое умножение, порядок i, j, k.
Но тут кроется проблема! Во внутреннем цикле k происходит обращение к
элементам матрицы B по столбцам. Поскольку в C++ матрицы хранятся в
Row-major order, каждый шаг k это прыжок в памяти на размер всей строки.
Результат для целевых матриц 64x64: 3.63 FLOPS/такт.
Результат для больших матриц 2048x2048: 0.16 FLOPS/такт.
Вывод: Процессор 95% времени простаивает в ожидании данных из RAM. Низкая
пространственная локальность делает этот метод непригодным для больших данных.

ЭТАП 2: ИЗМЕНЕНИЕ ПОРЯДКА ЦИКЛОВ (IKJ) И АВТО-ВЕКТОРИЗАЦИЯ
---------------------------------------------------------
Смена порядка циклов на i -> k -> j.
По сути мы выполнили транспонирование второй матрицы. Теперь внутренний цикл j идет
строго последовательно по строке матрицы B и строке результата result.
Результат для целевых матриц 64x64: 12.18 FLOPS/такт, это пиковая скорость в L1.
Результат для больших матриц 2048x2048: 3.28 FLOPS/такт - ускорение в 20 раз,
по сравнению с наивной версией.
Вывод: Это обеспечило идеальную пространственную локальность. Процессор
загружает целую кэш-линию 64 байта и эффективно обрабатывает её. Компилятор
GCC смог автоматически применить векторные инструкции SIMD, так как данные
лежат подряд.

ЭТАП 3: РУЧНАЯ ВЕКТОРИЗАЦИЯ (AVX INTRINSICS)
--------------------------------------------
Использование интринсиков AVX2 (_mm256_loadu_ps, _mm256_fmadd_ps) с целью прямого
управления 256-битными регистрами, т.е. обработка 8 float за такт
и использование FMA (Fused Multiply-Add) для совмещения умножения и сложения.
В результате получили стабильную работу на уровне авто-векторизации, но с более жестким
контролем над темпом выдачи инструкций.

ЭТАП 4: РЕГИСТРОВЫЙ АККУМУЛЯТОР (IMPROVED AVX)
----------------------------------------------
Под термином регистровый аккумулятор понимается использование векторного регистра
процессора для накопления промежуточной суммы произведения матриц без обращения
к оперативной памяти во внутреннем цикле. Это позволяет сократить количество инструкций
LOAD/STORE и снизить нагрузку на подсистему памяти, обеспечивая максимальный темп
работы вычислительных блоков FMA.

Здесь попытка минимизировать количество операций записи в память (STORE operations).
Один элемент матрицы A загружается в регистр один раз (_mm256_set1_ps) и
используется для накопления суммы прямо в векторном регистре.
Результат для целевых матриц 64x64: 11.91 FLOPS/такт, тоже высокая скорость в L1.
Результат для больших матриц 2048x2048: 0.64 FLOPS/такт, а это провал.
Вывод: Этот этап выявил важный архитектурный нюанс. Для эффективной работы
аккумулятора пришлось использовать порядок IJK. На малых матрицах внутри L1 кэша
это дало огромную скорость. На больших матрицах промахи кэша из-за порядка
IJK полностью нивелировали выгоду от экономии регистровых операций.

ЭТАП 5: БЛОЧНОЕ УМНОЖЕНИЕ (AVX TILED)
------------------------------------------------
Здесь уже сочетание порядка IKJ и разбиения матриц на блоки 64x64
с целью обеспечения временной локальности.
По сути мы заставляем процессор полностью завершить вычисления в рамках
маленького блока данных, прежде чем двигаться дальше.
Параметры: BS=64 выбрано исходя из размера L1 кэша i7-1260P (48 KB).
Три блока float 64x64 занимают (64(elems) * 64(elems) * 4(bytes)) * 3(matrix) = 48 KB.
Результат для целевых матриц 64x64: 8.35 FLOPS/такт,
неплохая скорость в L1, но хуже, чем было.
Результат для больших матриц 2048x2048: 4.41 FLOPS/такт - победитель среди CPU.
Вывод: Тайлинг позволил обмануть медленную оперативную память. Данные
загружаются в L1 один раз и многократно переиспользуются внутри блока. Это
единственный способ достичь высокой производительности на матрицах, размер
которых кратно превышает объем кэш-памяти.

ИТОГОВАЯ ИЕРАРХИЯ ОПТИМИЗАЦИЙ:
1. Алгоритмическая (Порядок циклов) -> Базовый уровень.
2. Векторная (SIMD / AVX2) -> Параллелизм данных.
3. Микроархитектурная (FMA / Unrolling) -> Параллелизм инструкций.
4. Системная (Tiling / Blocking) -> Управление иерархией памяти.

Комбинация всех четырех уровней позволила поднять производительность CPU
с 0.16 до 4.41 FLOPS/такт на больших данных и с 3.63 до 12.18 на целевых данных,
обеспечив 27-кратное и 3.4-кратное ускорение соответственно.

МЕТОДОЛОГИЯ ИССЛЕДОВАНИЯ
---------------------------
Для тестирования выбраны несколько сценариев, последние 2 дополнительно:
1. Small (64x64): Весь набор данных (48 KiB) полностью помещается в L1d кэш.
2. Medium (1024x1024): Данные уже полностью помещаются только в L3 кэш.
3. Large (2048x2048): Объем данных (~48 MiB) значительно превышает L3 кэш,
   что заставляет тестировать подсистему памяти и эффективность тайлинга.

Измерения проводились в тактах процессора с использованием инструкции __rdtsc().
На аппаратном уровне он разворачивается в инструкцию RDTSC (Read Time-Stamp Counter),
которая считывает содержимое 64-битного регистра TSC
(Time Stamp Counter, считают количество тактов с момента последнего сброса).

Особенности реализации на Intel i7-1260P:
- В данной архитектуре счетчик TSC инкрементируется с постоянной
  номинальной частотой, не зависящей от Turbo Boost или энергосберегающих
  состояний ядер. Это гарантирует воспроизводимость замеров.
- Низкий Overhead: Вызов __rdtsc() занимает ~20-40 тактов, что позволяет
  измерять даже малые участки кода (64x64) с минимальной погрешностью.
- Единица измерения: Все результаты приведены в cycles и
  производительности FLOPS/cycle.

АНАЛИЗ РЕЗУЛЬТАТОВ
---------------------
1. Проблема Naive (IJK):
При больших размерах матриц порядок циклов IJK приводит к чтению матрицы B
по столбцам. Это вызывает постоянные промахи кэша, так как
каждый шаг по K это прыжок через всю строку размером 8 КБ. Производительность
составляет менее 1% от пика, а именно 0.16 FLOPS/такт.

2. Преимущество Блочного умножения:
Версия 3.2 показала лучший результат на CPU - 4.41 FLOPS/такт, обойдя
авто-векторизацию компилятора. Выбран размер блока BS=64, что соответствует
16 КБ на матрицу, суммарно 48 КБ, что идеально ложится в объем кэша L1d
процессора i7-1260P. Это минимизирует трафик между L1 и RAM.

3. Анализ провала Improved версии (IJK + Accumulator):
Версия 3.1, показавшая отличные результаты на 64x64, полностью провалилась
на 2048x2048, показав 0.64 FLOPS/такт. Это доказывает, что на больших данных
локальность памяти важнее использования регистровых аккумуляторов.

4. OpenCL GPU:
Видеокарта показала в ~2.2 раза большую производительность, чем лучший
метод на CPU. Это обусловлено высокой пропускной способностью памяти GPU
и массивным параллелизмом, который нивелирует задержки доступа к данным.

ФУНКЦИОНАЛЬНАЯ КОРРЕКТНОСТЬ
-----------------------------------------
При сравнении результатов Naive и AVX/GPU на больших матрицах (2048)
наблюдается расхождение (в коде обозначено Mismatch) в пределах 0.1-0.5%.
Причины:
1. Порядок суммирования: Naive - последовательно, AVX - блочно по 8 штук.
2. Инструкции FMA (Fused Multiply-Add): Выполняют умножение и сложение за
   один этап с одним округлением, что точнее стандартных операций.
3. Ограничения fp32: При накоплении суммы из 2048 элементов, ошибка
   округления в 7 значащих цифрах float становится заметной, но все еще
   укладывается в относительную ошибку.
На малых матрицах 64x64 результаты идентичны SUCCESS, что подтверждает
логическую верность всех реализаций.

ВЫВОДЫ
---------
1. Порядок обхода памяти IKJ/transponed vs IJK является критическим фактором,
   дающим ускорение в ~20 раз на больших данных.
2. Использование интринсиков AVX2/FMA без учета иерархии кэша не дает
   преимущества. Эффективность достигается только при сочетании векторизации
   и тайлинга.
3. Размер блока BS=64 является оптимальным для архитектуры исследуемого процессора,
   так как позволяет удерживать рабочий набор данных в кэше L1d = 48 KB.
4. GPU является наиболее эффективным инструментом для матричного умножения
   больших размерностей, в то время как на малых данных 64x64 CPU
   выигрывает за счет отсутствия накладных расходов на передачу данных.

Полная загрузка векторных регистров при помощи AVX-инструкций

Был написан мини-кернел AVX 3.3, соединяющий в себе преимущества предыдущих версий, а также дополненный
загрузкой максимально возможного числа векторных регистров.
В случае процессора Intel 1260P отсутствуют 512-битные регистры, 256-битных 16 штук, загрузили 14 из них.

Просадка производительности на 1024x1024 до 35% связана с Memory Wall:
необходимые данные не помещаются в кэш полностью и их необходимо подгружать.
При этом отличие в результатах начинает проявляться уже на матрицах 256x256,
результаты согласованы с автовекторизированной версией.
Отличным был бы результат загрузки ядра более 85%. Свои результаты я могу связать с расходами на
branches и циклы. Очень вероятно, что алгоритм будет работать быстрее, если будет реализация для матриц
одного определенного размера.
